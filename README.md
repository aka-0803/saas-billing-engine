## Request Flow

```
HTTP Request
     │
     ▼
[RateLimitGuard]    ← GLOBAL — per-tenant sliding 60s window (plan-aware, Redis-backed)
     │
     ▼
[JwtAuthGuard]      ← per controller — Bearer token → req.user { userId, tenantId }
     │
     ▼
[ValidationPipe]    ← GLOBAL — class-validator, whitelist + transform
     │
     ▼
[Controller → Service → Prisma → MySQL]
     │
     ▼ (post-response)
[UsageInterceptor]  ← GLOBAL — auto-increments tenant usage +1 per authenticated request

── Invoice Pipeline ──────────────────────────────────────────────
BillingService.generateInvoice()
     │
     ▼
[Invoice row created]   ← synchronous, pdf_url = null
     │
     ▼
[BullMQ job enqueued]   ← non-blocking, stored in Redis (3 retries, exponential backoff)
     │
     ▼ (background worker)
[InvoiceProcessor]      ← builds PDF buffer → uploads to S3 → saves pdf_url on Invoice

── Invoice Download ──────────────────────────────────────────────
GET /billing/invoice/:id/download
     │
     ▼
[S3Service.getPresignedUrl()]   ← 15-min time-limited URL
     │
     ▼
{ download_url }    ← client downloads directly from S3
```

## Features

- **Multi-tenancy** — all data scoped per tenant; JWT payload carries `tenantId`
- **Usage-based billing** — overage pricing: `total = plan.price + max(0, usage − limit) × $2/unit`
- **Async invoice PDFs** — generated by a BullMQ worker, uploaded to S3, never blocks the HTTP response
- **Pre-signed download URLs** — clients download invoices directly from S3 (15-min TTL)
- **Subscription lifecycle** — create, cache (Redis 5 min), increment usage, renew (invoice → roll dates → reset)
- **Payment simulation** — idempotency-key deduplication via Redis (24 h window)
- **Plan-aware rate limiting** — per-tenant sliding 60s window from `plan.rate_limit_per_minute`
- **Global usage tracking** — `UsageInterceptor` writes a `UsageRecord` after every authenticated request
- **Daily billing cron** — auto-renews subscriptions where `end_date ≤ today` at midnight
- **CI/CD** — GitHub Actions deploys to EC2 on every push to `main`

## Deployment (AWS EC2)

The app runs on EC2 via Docker Compose. Deployment is fully automated — every push to `main` triggers GitHub Actions which SSHs into EC2, pulls the latest code, and runs `docker compose up -d --build`.

### First-time EC2 setup

```bash
# Install Docker
sudo dnf install -y docker git
sudo service docker start
sudo usermod -aG docker ec2-user
sudo systemctl enable docker

# Install Docker Buildx
mkdir -p ~/.docker/cli-plugins
curl -SL https://github.com/docker/buildx/releases/download/v0.17.1/buildx-v0.17.1.linux-amd64 \
  -o ~/.docker/cli-plugins/docker-buildx
chmod +x ~/.docker/cli-plugins/docker-buildx

# Install Docker Compose plugin
sudo mkdir -p /usr/local/lib/docker/cli-plugins
sudo curl -SL https://github.com/docker/compose/releases/latest/download/docker-compose-linux-x86_64 \
  -o /usr/local/lib/docker/cli-plugins/docker-compose
sudo chmod +x /usr/local/lib/docker/cli-plugins/docker-compose

# Clone and start
git clone https://github.com/aka-0803/saas-billing-engine.git
cd saas-billing-engine
cp .env.example .env && nano .env
docker compose up -d --build
```

### Restart procedure (EC2/RDS stopped to save cost)

1. Start RDS → wait ~2 min ("Available")
2. Start EC2 → wait ~1 min
3. EC2 Instance Connect → `cd saas-billing-engine && docker compose up -d`
4. Verify: `http://13.48.21.213:3000/api/docs`

## Key Design Decisions

**Why Interceptor for usage tracking (not Guard or Service)?**
Guards are binary (allow/block). Interceptors wrap the handler and only fire post-response — usage is only counted for completed requests. Errors are swallowed so tracking never breaks the response.

**Why soft delete (is_deleted int) not hard delete?**
Invoices and usage records reference tenants/users. Hard delete breaks referential integrity and audit trail.

**Why Redis for subscription cache?**
Subscription lookups happen on every rate-limit check and usage increment. TTL 300s significantly reduces DB load.

**Why processRenewal invoices BEFORE rolling dates?**
The invoice must reflect the period it covers. Rolling dates first would make the invoice show future dates against past usage.

**Why idempotency key on payments?**
Payment webhooks are retried by providers on network failure. Without deduplication the same webhook would double-pay.

**Why INVOICE_QUEUE constant in `billing.constants.ts`?**
If the constant lived in `billing.module.ts`, the circular import (`BillingModule` → `BillingService` → queue token → `BillingModule`) would crash NestJS at startup.

**Why MySQL on RDS, not in Docker Compose?**
If EC2 dies, RDS data survives. Running MySQL in Compose ties DB lifecycle to the instance.
